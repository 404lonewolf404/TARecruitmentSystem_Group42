# 需求文档

## 简介

TA招聘系统是为北京邮电大学国际学院开发的网页应用程序，用于管理助教招聘。该系统将取代现有的电子表格流程，提供结构化的工作流程：TA可以申请职位，模块组织者（MO）可以发布和管理职位，管理员可以监控工作量分配。

## 术语表

- **TA**: 助教 - 申请助教职位的学生
- **MO**: 模块组织者 - 发布TA职位并选择申请者的教师
- **Admin**: 管理员 - 监控整体TA工作量的系统管理员
- **System**: TA招聘系统网页应用
- **Position**: 职位 - MO创建的助教工作岗位
- **Application**: 申请 - TA对特定职位的申请请求
- **Profile**: 个人资料 - 包含个人详细信息和资质的用户账户信息
- **Session**: 会话 - 已认证用户与系统的活动连接
- **CSV_Store**: CSV存储 - 使用CSV格式的基于文件的数据存储系统

## 需求

### 需求1：用户注册和认证

**用户故事：** 作为用户（TA或MO），我想注册账号并登录，以便我可以访问系统功能。

#### 验收标准

1. 当新用户提供有效的注册信息（姓名、邮箱、密码、角色）时，系统应创建新用户账号
2. 当用户提供已存在的邮箱时，系统应拒绝注册并显示错误消息
3. 当已注册用户提供正确的凭证时，系统应验证用户并创建会话
4. 当用户提供错误的凭证时，系统应拒绝登录尝试并显示错误消息
5. 当已认证用户请求登出时，系统应终止会话并重定向到登录页面
6. 当用户尝试访问受保护页面而未认证时，系统应重定向到登录页面

### 需求2：TA个人资料管理

**用户故事：** 作为TA，我想创建和管理我的个人资料，以便MO可以评估我的资质。

#### 验收标准

1. 当TA创建个人资料时，系统应存储TA的姓名、邮箱和技能
2. 当TA更新其个人资料时，系统应将更改持久化到CSV存储
3. 当TA查看其个人资料时，系统应显示所有当前的个人资料信息
4. 系统应验证必填的个人资料字段（姓名、邮箱）不为空

### 需求3：MO职位管理

**用户故事：** 作为MO，我想发布TA职位，以便合格的TA可以申请。

#### 验收标准

1. 当MO使用有效信息（标题、描述、要求、工时）创建职位时，系统应存储该职位
2. 当MO查看其职位时，系统应显示该MO创建的所有职位
3. 当MO删除职位时，系统应移除该职位及所有相关申请
4. 系统应验证必填的职位字段（标题、描述、工时）不为空
5. 当创建职位时，系统应为其分配唯一标识符

### 需求4：TA职位申请

**用户故事：** 作为TA，我想浏览可用职位并提交申请，以便我可以被考虑担任TA角色。

#### 验收标准

1. 当TA查看职位列表时，系统应显示所有可用职位及其详细信息
2. 当TA申请职位时，系统应创建将TA与职位关联的申请记录
3. 当TA尝试重复申请同一职位时，系统应拒绝重复申请
4. 当TA查看其申请时，系统应显示其申请的所有职位及当前状态
5. 当TA撤回申请时，系统应移除申请记录

### 需求5：MO申请审核

**用户故事：** 作为MO，我想审核我职位的申请并选择候选人，以便我可以雇用合格的TA。

#### 验收标准

1. 当MO查看其职位的申请时，系统应显示所有申请的TA及其个人资料信息
2. 当MO为职位选择TA时，系统应将申请状态更新为"已选中"
3. 当MO选择TA时，系统应将该职位的所有其他申请更新为"已拒绝"
4. 当申请状态改变时，系统应将更改持久化到CSV存储

### 需求6：管理员工作量监控

**用户故事：** 作为管理员，我想查看所有TA的工作量，以便我可以确保工作时间的公平分配。

#### 验收标准

1. 当管理员请求工作量报告时，系统应显示所有TA及其总分配工时
2. 当计算工作量时，系统应汇总TA申请状态为"已选中"的所有职位的工时
3. 系统应在工作量报告中显示TA姓名、邮箱和总工时

### 需求7：数据持久化

**用户故事：** 作为系统操作员，我想将所有数据存储在CSV文件中，以便系统可以在没有数据库的情况下运行。

#### 验收标准

1. 当用户数据改变时，系统应将更新的数据写入users.csv
2. 当职位数据改变时，系统应将更新的数据写入positions.csv
3. 当申请数据改变时，系统应将更新的数据写入applications.csv
4. 当系统启动时，系统应从CSV文件加载所有数据
5. 如果CSV文件不存在，系统应创建带有适当标题的空文件

### 需求8：会话管理

**用户故事：** 作为用户，我想维持我的登录会话，以便我不必为每个页面重新认证。

#### 验收标准

1. 当用户成功登录时，系统应创建包含用户信息的会话
2. 当会话处于活动状态时，系统应允许访问与角色相应的页面
3. 当会话过期或用户登出时，系统应要求重新认证
4. 系统应在会话中存储用户角色以强制执行访问控制
